# 라벨 이미지 확장 검토

## 1. 문서 목적

이 문서는 `thumbnail-maker`에 **하단 라벨 이미지 오버레이 기능**을 추가할 때,

- 기존 `thumbnail-maker`를 확장하는 것이 좋은지
- 별도 스킬 또는 별도 서브에이전트로 분리하는 것이 좋은지

를 검토하고, 현재 프로젝트에 맞는 권장안을 정리한 문서입니다.

---

## 2. 이번에 추가하려는 기능의 성격

요청하신 기능은 "라벨 생성"이 아니라, 이미 준비된 라벨 이미지를 **원본 이미지 위에 위치시키는 오버레이 기능**입니다.

즉 작업의 본질은 여전히 동일합니다.

- 원본 이미지 받기
- 정사각형 썸네일 만들기
- 브랜드 로고 넣기
- 추가 배지/라벨 넣기
- 저장 규칙에 따라 파일 저장하기

따라서 이 기능은 완전히 새로운 작업이라기보다,  
**기존 썸네일 생성 파이프라인의 옵션 확장**에 가깝습니다.

---

## 3. 먼저 결론

현재 단계에서는 **기존 `thumbnail-maker`를 확장하는 것이 가장 적절합니다.**

별도 스킬이나 별도 서브에이전트로 분리하는 것은 지금은 과합니다.

---

## 4. 왜 `thumbnail-maker` 확장이 좋은가

### 이유 1. 작업 목적이 바뀌지 않는다

지금도 목적은 "상품 썸네일 생성"입니다.

라벨이 들어가도 결과물은 여전히 썸네일이고,  
입력도 여전히 원본 이미지이며,  
저장 위치와 출력 규격도 동일합니다.

즉,

- 별도 산출물 유형이 생기는 것이 아니고
- 별도 검수 체계가 필요한 것도 아니고
- 전혀 다른 제작 프로세스가 생기는 것도 아닙니다

그래서 `thumbnail-maker`의 옵션 확장으로 다루는 것이 구조적으로 자연스럽습니다.

### 이유 2. 현재 구현 구조와 잘 맞는다

현재 `thumbnail-maker`는 이미 아래 요소를 가지고 있습니다.

- 정사각형 크롭
- 로고 오버레이
- 설정 파일 기반 값 관리
- 저장 경로 규칙

라벨 이미지 오버레이는 여기서 크게 벗어나지 않습니다.

추가되는 것은 사실상 아래뿐입니다.

- 라벨 자산 경로
- 라벨 종류
- 라벨 위치
- 복수 라벨 배치 규칙

즉, **현재 스크립트와 설정 구조를 확장하기 쉬운 종류의 변경**입니다.

### 이유 3. 디자이너 입장에서 사용성이 단순하다

디자이너 입장에서는 아래 둘 중 무엇이 더 쉬운지가 중요합니다.

1. 썸네일 만들기 요청 시 옵션으로 라벨 추가
2. 라벨용 별도 스킬/에이전트를 따로 선택

현재는 1번이 훨씬 직관적입니다.

예:

- 기본 썸네일 생성
- 무료배송 라벨 추가
- 골라담아할인 라벨 추가
- 두 라벨 모두 추가

이 정도는 같은 썸네일 작업 안에서 옵션으로 다루는 것이 가장 단순합니다.

---

## 5. 언제 별도 스킬로 분리하는 것이 맞을까

아래 조건이 생기면 그때는 **별도 스킬 분리**를 검토할 수 있습니다.

### 분리 신호

- 라벨 종류가 크게 늘어난다
- 라벨마다 배치 규칙이 많이 다르다
- 채널별 라벨 정책이 달라진다
- 라벨 조합 로직이 복잡해진다
- 썸네일 외 카드형 이미지, 배너형 이미지까지 같은 라벨 체계가 재사용된다

이 경우에는 `thumbnail-maker` 안에 모든 라벨 규칙을 넣기보다,  
예를 들어 `thumbnail-label-overlay` 같은 보조 스킬로 분리하는 것이 나아질 수 있습니다.

---

## 6. 언제 별도 서브에이전트로 분리하는 것이 맞을까

아래 수준까지 가면 그때는 **별도 서브에이전트**를 고려할 수 있습니다.

### 분리 신호

- 라벨 포함 썸네일이 일반 썸네일과 완전히 다른 결과물로 운영된다
- 입력 항목이 달라진다
- 검수 기준이 크게 달라진다
- 전용 작업자가 따로 사용할 정도로 워크플로우가 분리된다
- 하단 라벨형 썸네일이 외부몰 전용 포맷처럼 독립된 산출물 유형이 된다

예를 들어:

- 일반 썸네일용 `thumbnail_maker`
- 외부몰 라벨형 썸네일 전용 `marketplace_badge_thumbnail_maker`

처럼 목적 자체가 갈라질 때만 별도 서브에이전트가 타당합니다.

현재 요구사항은 아직 그 단계는 아닙니다.

## 7. 현재 프로젝트에 대한 권장안

### 권장안

**1차는 `thumbnail-maker` 확장**

구체적으로는 아래처럼 가는 것이 좋습니다.

1. `assets/`에 라벨 이미지 추가
2. `config/thumbnail.toml`에 라벨 관련 구성값 추가
3. `SKILL.md`에 라벨 오버레이 규칙 추가
4. `scripts/create_thumbnail.swift`를 라벨 오버레이까지 처리하도록 확장

### 권장 구조 예시

```text
/.agents/skills/thumbnail-maker
  /assets
    pulmuone-shop.svg
    free-shipping-label.svg
    mix-and-match-discount-label.svg
```

## 8. 라벨 기능은 어떻게 설계하면 좋은가

현재 요구사항을 기준으로 보면, 스킬에 아래 개념을 추가하는 것이 적절합니다.

### 추가할 입력 개념

- 라벨 사용 여부
- 라벨 종류
- 복수 라벨 허용 여부

### 추가할 구성값

- label_enabled
- label_asset_map
- label_anchor
- label_left
- label_bottom
- label_gap

### 예시 설정 방향

```toml
label_enabled = false
label_anchor = "left-bottom"
label_left = 60
label_bottom = 60
label_gap = 60

[label_asset_map]
free_shipping = ".agents/skills/thumbnail-maker/assets/free-shipping-label.svg"
mix_and_match_discount = ".agents/skills/thumbnail-maker/assets/mix-and-match-discount-label.svg"
```

## 9. 배치 규칙은 단순하게 시작하는 것이 좋다

처음부터 복잡하게 하지 말고 아래처럼 고정 규칙으로 시작하는 것을 권장합니다.

### 1차 권장 규칙

- 라벨은 썸네일 하단에 배치
- 라벨은 원본 이미지 위에 오버레이
- 라벨 크기는 원본 비율 유지
- 라벨이 1개면 좌측 하단 기준으로 배치
- 라벨이 2개면 좌측 하단에서 시작해 가로로 나란히 배치
- 라벨 간 간격은 설정값으로 관리

이 정도면 구현 난이도와 운영 난이도가 모두 낮습니다.

## 10. 이번 요청 기준 배치 규칙 상세

이번 요청에서는 라벨 배치 규칙을 아래처럼 **고정 규칙**으로 정리하는 것이 좋습니다.

### 기본 전제

- 라벨은 요청이 있을 때만 추가한다
- 요청이 없으면 라벨을 추가하지 않는다
- 라벨은 모두 썸네일 이미지 위에 오버레이한다
- 기준 좌표는 **최종 600x600 결과 이미지 기준**으로 계산한다

### 라벨이 1개만 요청된 경우

요청 라벨이 `무료배송`이든 `골라담아할인`이든 동일하게 처리합니다.

- 좌측 여백: `60px`
- 하단 여백: `60px`
- 배치 기준: 좌측 하단

즉, 라벨 1개일 때는 항상 **왼쪽 아래에 붙는 방식**으로 통일하는 것이 좋습니다.

### 라벨이 2개 모두 요청된 경우

두 라벨이 모두 요청되면 아래 순서를 고정합니다.

1. `무료배송` 라벨을 좌측 하단에 배치
2. `골라담아할인` 라벨을 같은 Y축에 배치
3. `골라담아할인` 라벨의 X 시작점은 `무료배송 라벨의 오른쪽 끝 + 60px`

즉, 2개일 때는 아래 규칙입니다.

- 첫 라벨 기준 위치: left `60px`, bottom `60px`
- 두 번째 라벨 간격: `60px`
- 두 라벨의 하단 정렬선은 동일

### 정리된 배치 개념

- 1개 요청: `left = 60`, `bottom = 60`
- 2개 요청:
  - free_shipping: `left = 60`, `bottom = 60`
  - mix_and_match_discount: `left = free_shipping.right + 60`, `bottom = 60`

## 11. `SKILL.md`에는 어떻게 정리하면 좋은가

`SKILL.md`에는 아래 3가지를 분리해서 적는 것이 좋습니다.

### 1) 입력 규칙

예시 문장:

```md
## Label Input Rules

- Labels are optional.
- If the user does not request labels, do not add any label overlay.
- Supported labels are:
  - free_shipping
  - mix_and_match_discount
```

### 2) 배치 규칙

예시 문장:

```md
## Label Placement Rules

- Compute all label positions against the final thumbnail canvas.
- If exactly one label is requested, place that label at:
  - left: 60px
  - bottom: 60px
- If both labels are requested:
  - place `free_shipping` at left 60px, bottom 60px
  - place `mix_and_match_discount` on the same baseline
  - start `mix_and_match_discount` at `free_shipping.right + 60px`
- Preserve each label image aspect ratio.
```

### 3) 잠금 규칙

예시 문장:

```md
## Locked Label Behavior

- Do not add labels unless the user explicitly requests them.
- Do not center a single label by default.
- Do not reorder the two-label layout by default.
- When both labels are requested, `free_shipping` must stay left of `mix_and_match_discount`.
```

이렇게 쓰면 스킬 문서만 읽어도,  
**라벨이 선택 기능인지**,  
**한 개일 때 어디에 놓는지**,  
**두 개일 때 어떤 순서와 간격으로 놓는지**가 바로 이해됩니다.

## 13. 최종 권장 결론

현재 요구사항 기준으로는 아래가 가장 적절합니다.

1. 별도 서브에이전트는 만들지 않는다
2. 별도 스킬도 아직 만들지 않는다
3. 기존 `thumbnail-maker`를 확장한다
4. 라벨 자산, 라벨 설정값, 라벨 배치 규칙을 기존 썸네일 생성 구조에 포함한다

즉, 지금 단계의 라벨 기능은 **"썸네일 생성 옵션 확장"** 으로 보는 것이 맞습니다.
