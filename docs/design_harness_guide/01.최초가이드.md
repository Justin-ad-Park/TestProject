# 디자인 작업용 GPT 하네스 엔지니어링 최초 가이드

## 1. 이 프로젝트는 무엇을 하려는가

이 프로젝트의 목적은 **디자인팀이 반복적으로 만드는 이미지를 GPT 기반 에이전트로 안정적으로 생성**할 수 있도록,  
요청 방식과 제약사항을 체계적으로 관리하는 것입니다.

단순히 "이미지 잘 뽑는 프롬프트"를 모으는 것이 아니라, 아래를 함께 관리하는 운영 체계를 만드는 것이 핵심입니다.

- 어떤 종류의 작업을 어떤 에이전트가 담당하는지
- 각 작업에서 꼭 지켜야 하는 디자인 제약이 무엇인지
- 요청자가 어떤 형식으로 요청해야 결과 품질이 좋아지는지
- 결과물이 어긋났을 때 무엇을 개선해야 하는지
- 개선된 규칙을 어떻게 문서화하고 재사용할지

즉, 이 프로젝트는 **디자인 이미지 생성용 에이전트 운영 시스템**을 만드는 일입니다.

---

## 2. 왜 먼저 "하네스"부터 설계해야 하는가

이미지 생성 품질은 모델 자체보다도 아래 요소에 크게 좌우됩니다.

- 요청 내용이 얼마나 구체적인가
- 작업 종류별 제약이 분리되어 있는가
- 실패 사례가 다음 요청에 반영되는가
- 팀이 같은 기준으로 요청하고 검수하는가

그래서 처음부터 하나의 범용 에이전트로 모든 작업을 처리하기보다,  
**업무 유형별로 역할을 나누고**,  
**각 에이전트의 요청 템플릿과 금지사항을 별도로 관리**하는 방식으로 시작하는 것이 좋습니다.

---

## 3. 가장 먼저 만들 에이전트 구조

처음부터 너무 세분화하기보다, 실제 디자인팀 업무 흐름 기준으로 4~5개 정도의 에이전트로 시작하는 것을 권장합니다.

### 권장 1차 에이전트 구성

| 에이전트 | 담당 작업 | 핵심 특징 |
| --- | --- | --- |
| 배너 에이전트 | 메인 배너, 띠배너, 프로모션 배너 | 강한 주목도, 빠른 메시지 전달, 배치 안정성 중요 |
| 상품상세 이미지 에이전트 | 상세페이지용 설명 이미지, 소재 강조 이미지 | 정보 전달력, 제품 맥락, 텍스트 삽입 여백 중요 |
| 이벤트 에이전트 | 시즌 이벤트, 타임딜, 쿠폰전 등 | 분위기 연출, 혜택 강조, 캠페인 톤 통일 중요 |
| 기획전 에이전트 | 테마전, 카테고리전, 브랜드전 | 여러 상품을 묶는 세계관/컨셉 정리 중요 |
| 외부몰 에이전트 | 오픈마켓/제휴몰 배너, 썸네일, 상품 이미지 | 채널별 제약, 단순명료함, 심사 리스크 회피 중요 |

### 중요한 원칙

**에이전트는 "디자인 스타일"이 아니라 "업무 목적" 기준으로 나눕니다.**

예를 들어 배너와 상품상세 이미지는 둘 다 예쁘게 만들어야 하지만,

- 배너는 클릭 유도
- 상세 이미지는 정보 전달

이라는 목적 차이가 커서 같은 규칙으로 운영하면 품질이 흔들리기 쉽습니다.

## 4. 각 에이전트마다 반드시 관리해야 하는 5가지

에이전트를 만들 때는 프롬프트 한 장만 만들지 말고, 아래 5가지를 세트로 관리해야 합니다.

### 1) 역할 정의

이 에이전트가 어떤 작업을 담당하는지 명확히 적습니다.

예시:

> 이 에이전트는 쇼핑몰 프로모션 배너용 이미지 생성을 담당한다.  
> 첫 화면 주목도, 카피 배치 여백, 상품 또는 혜택 메시지 강조를 우선한다.

### 2) 입력 항목

요청자가 무엇을 넣어야 하는지 표준화합니다.

예시:

- 작업 종류
- 노출 위치
- 배너 사이즈
- 전달할 핵심 메시지
- 포함할 상품/브랜드
- 원하는 분위기
- 금지 요소
- 참고 레퍼런스

### 3) 출력 원칙

좋은 결과물의 기준을 정의합니다.

예시:

- 텍스트가 올라갈 영역이 충분히 비어 있어야 함
- 상품이 주인공일 경우 배경 연출이 과하지 않아야 함
- 할인/혜택 중심 배너는 메시지 가독성이 최우선이어야 함

### 4) 금지사항

실패를 줄이는 데 가장 중요한 영역입니다.

예시:

- 과도한 장식 요소 사용 금지
- 상품 원형을 왜곡하는 연출 금지
- 읽기 어려운 복잡한 배경 금지
- 외부몰 심사에서 문제될 표현 금지

### 5) 개선 로그

반드시 남겨야 합니다.

예시:

- "배너에서 텍스트 자리가 부족하게 나오는 문제가 반복됨"
- "외부몰 썸네일은 배경 연출보다 상품 분리가 더 중요함"
- "식품 상세 이미지는 과장된 연기/광택 표현이 실제 상품감과 어긋남"

이 로그가 쌓여야 에이전트가 팀의 실제 업무 기준에 맞게 진화합니다.

## 5. 실제 운영은 "좋은 요청서"에서 시작된다

처음에는 모델 성능보다 **요청서 품질**이 결과를 더 많이 좌우합니다.  
따라서 각 에이전트별로 요청 양식을 먼저 만드는 것이 좋습니다.

### 요청서에 꼭 들어가야 할 기본 항목

| 항목 | 설명 |
| --- | --- |
| 작업명 | 어떤 작업인지 한 줄로 설명 |
| 사용처 | 메인, 카테고리, 상세, 외부몰 등 |
| 목표 | 클릭 유도, 정보 전달, 이벤트 고지 등 |
| 핵심 메시지 | 사용자에게 가장 먼저 보이길 원하는 내용 |
| 필수 오브젝트 | 상품, 브랜드 요소, 배경 소재 등 |
| 원하는 무드 | 고급, 산뜻, 시즌감, 실용적 등 |
| 금지사항 | 피해야 할 색, 구성, 표현 방식 |
| 참고자료 | 레퍼런스 링크, 기존 디자인, 브랜드 톤 |

### 요청서 예시

```md
- 작업명: 여름 시즌 과일 기획전 히어로 배너
- 사용처: 앱 메인 상단 배너
- 목표: 여름 제철 과일 기획전 유입 유도
- 핵심 메시지: 시원하고 신선한 제철 과일 모음
- 필수 오브젝트: 수박, 복숭아, 청포도
- 원하는 무드: 밝고 청량함, 프리미엄보다는 신선함 중심
- 금지사항: 과한 3D 느낌, 난잡한 소품, 어두운 톤
- 참고자료: 지난 시즌 메인 배너 톤, 경쟁사 여름 과일전 레퍼런스
```

## 6. 처음부터 완벽한 프롬프트를 만들려 하지 말 것

초기에는 아래 방식이 가장 현실적입니다.

### 추천 방식: 초안 -> 테스트 -> 수정 -> 축적

1. 에이전트별 초안 프롬프트를 만든다.
2. 실제 업무와 비슷한 요청 5~10개로 테스트한다.
3. 실패 패턴을 기록한다.
4. 금지사항과 출력 원칙을 보강한다.
5. 팀 공용 문서에 개선 내용을 누적한다.

처음부터 정답을 만들기보다,  
**실패를 빠르게 수집하고 규칙으로 바꾸는 체계**를 만드는 것이 더 중요합니다.

---

## 7. 디자인팀 기준으로 꼭 정해야 하는 운영 원칙

프로젝트 초기에 아래 원칙을 합의해 두면 품질 관리가 쉬워집니다.

### 원칙 1. "예쁜 이미지"보다 "업무 목적 달성"을 우선한다

- 배너는 클릭 유도
- 상세 이미지는 정보 전달
- 이벤트 이미지는 행사 인지
- 외부몰 이미지는 심사 적합성과 상품 명확성

이미지의 미감은 중요하지만, **채널과 목적에 맞지 않으면 실패**입니다.

### 원칙 2. 에이전트별 금지사항은 별도로 관리한다

배너에서 허용되는 연출이 상세페이지에서는 방해가 될 수 있고,  
자사몰에서 괜찮은 표현이 외부몰에서는 제한될 수 있습니다.

그래서 공통 프롬프트 하나로 묶지 말고,  
**에이전트별 제약 문서를 분리**하는 것이 좋습니다.

### 원칙 3. 사람의 검수 기준을 문장으로 남긴다

디자이너가 결과를 보고 "뭔가 애매하다"라고 느꼈다면,  
그 판단을 문장으로 바꿔야 다음부터 개선할 수 있습니다.

예시:

- "텍스트를 올리면 답답해지는 구도는 피한다"
- "상품보다 배경 소품이 먼저 보이면 실패로 본다"
- "식품은 지나치게 비현실적인 질감보다 실제 구매욕을 높이는 자연스러움이 중요하다"

### 원칙 4. 레퍼런스 없이도 동작하는 요청 구조를 만든다

좋은 레퍼런스가 있으면 좋지만, 매번 충분하지는 않습니다.  
따라서 요청서만으로도 일정 수준 이상의 결과가 나오도록 입력 구조를 설계해야 합니다.

## 8. 프로젝트를 시작할 때 가장 현실적인 문서 구조

처음에는 복잡하게 시작하지 말고 아래 정도면 충분합니다.

```text
/docs
  /design_harness_guide
    01.최초가이드.md
    02.에이전트구조안.md
    03.공통요청서템플릿.md
    04.검수체크리스트.md
    05.개선로그운영방식.md
```

이후 에이전트별 운영이 시작되면 아래처럼 확장할 수 있습니다.

```text
/docs
  /agents
    /banner
    /detail
    /event
    /promotion
    /marketplace
```

## 9. 지금 바로 해야 할 첫 작업 우선순위

처음부터 많은 것을 만들 필요는 없습니다. 아래 순서로 시작하는 것을 권장합니다.

### 1단계

**에이전트 구분부터 확정**

- 배너
- 상품상세
- 이벤트
- 기획전
- 외부몰

### 2단계

**공통 요청서 템플릿 작성**

누가 요청해도 빠지지 않게 기본 입력 항목을 통일합니다.

### 3단계

**에이전트별 금지사항 초안 작성**

실패를 줄이기 위해 가장 먼저 필요한 문서입니다.

### 4단계

**검수 체크리스트 작성**

결과물을 어떤 기준으로 통과/수정할지 정의합니다.

### 5단계

**테스트용 샘플 작업 5~10건 선정**

실제 업무에 가까운 사례로 테스트해야 개선 포인트가 명확해집니다.

---

## 10. 추천하는 첫 운영 방식

초기에는 아래처럼 단순하게 운영하는 것이 좋습니다.

### 운영 흐름

1. 디자이너가 요청서 작성
2. 해당 작업 유형의 에이전트에 요청
3. 결과물 검수
4. 실패 이유 기록
5. 에이전트 규칙 업데이트

### 핵심 포인트

- 결과물만 저장하지 말고 **왜 수정이 필요했는지** 남길 것
- 잘 나온 사례보다 **안 맞았던 사례**를 더 중요하게 다룰 것
- 에이전트 설명, 입력 항목, 금지사항, 검수 기준을 따로 관리할 것

## 11. 이 프로젝트의 성공 기준

이 프로젝트가 잘 되고 있는지는 아래 기준으로 판단할 수 있습니다.

- 요청서 품질이 팀 내에서 점점 표준화되는가
- 작업 유형별 실패 패턴이 문서화되는가
- 같은 요청에 대한 결과 편차가 줄어드는가
- 디자이너가 "어떻게 요청해야 잘 나오는지" 빠르게 학습하는가
- 새 작업자도 문서만 보고 일정 수준으로 사용할 수 있는가

즉, 성공 기준은 단발성 고품질 이미지 1장이 아니라,  
**팀이 재현 가능한 방식으로 결과 품질을 높여 가는 체계가 생기는지**입니다.

---

## 12. 한 줄 정리

이 프로젝트는 이미지 생성용 프롬프트를 만드는 일이 아니라,  
**디자인팀의 실제 제작 업무를 에이전트 단위로 분리하고, 요청서/제약사항/검수기준/개선로그를 축적하는 운영 시스템을 만드는 일**입니다.
