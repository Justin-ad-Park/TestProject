# AX Commerce Resource Management

23명 규모 조직에서 프로젝트와 운영/개선 업무를 한눈에 보기 위해서는 "모든 정보를 한 파일에 쌓는 방식"보다 아래 3계층 구조가 더 안정적이다.

1. 임원/리더용 포트폴리오 뷰
2. 월/분기 단위 리소스 뷰
3. 개인별 작업 원장

이 구조의 목표는 두 가지다.

- 지금 누가 어떤 프로젝트/운영 업무를 맡고 있는지 빠르게 파악
- 신규 업무를 누구에게 배정할 수 있는지 빠르게 판단

## 파일별 역할

### `dashboard/executive_overview.md`

- 전체 프로젝트 현황
- 운영/개선 핵심 업무 묶음
- 조직별 현재 가동률
- 신규 할당 가능 인원 후보

### `dashboard/team_capacity.md`

- 팀별 현재 투입률
- 팀별 여유/과부하 상태
- 팀별 우선순위 충돌

### `dashboard/member_gantt_overview.md`

- 전체 구성원을 구성원 순으로 보는 Gantt 뷰
- 임원/리더가 현재 배정 상태를 빠르게 파악
- 팀별 상세 Gantt로 내려가기 전 요약판 역할

### `quarterly/YYYY_Qx.md`

- 분기 단위 프로젝트 로드맵
- 프로젝트별 주요 참여자
- 운영/개선 업무의 큰 묶음

### `monthly/YYYY-MM.md`

- 실제 배정 및 일정 변경 반영
- 개인 단위 작업 배치
- 주차 단위 충돌 관리

### `members/<team>/<name>.md`

- 구성원 개인의 현재 담당 업무
- 보유 역량, 주력 도메인
- 향후 4~8주 배정 가능성

## 관리 원칙

### 1. 프로젝트와 운영 업무를 같은 레벨로 두지 않는다

- 프로젝트: `EPIC` 중심으로 관리
- 운영/개선: `TASK` 묶음을 운영 테마 단위로 관리

예시:

- 프로젝트: `B2C 장바구니 개편`
- 운영 테마: `주문/결제 안정화`, `전시 운영 개선`, `사내몰 권한/정산 개선`

운영/개선 업무를 개별 TASK 그대로 상단 대시보드에 노출하면 너무 커진다. 상단에서는 운영 테마로 묶고, 상세 파일에서 TASK를 본다.

### 2. "사람 기준"과 "업무 기준"을 둘 다 유지한다

- 업무 기준: 어떤 프로젝트/운영 묶음이 진행 중인가
- 사람 기준: 각 인원이 무엇을 맡고 있는가

둘 중 하나만 있으면 배정이 어려워진다.

### 3. 배정 단위는 10% 또는 0.5일 단위로 단순화한다

정교한 시간 추정보다, 빠르게 비교 가능한 배정 단위가 중요하다.

예시:

- 20%: 유지보수성 업무
- 40%: 중간 규모 개선 과제
- 70% 이상: 메인 프로젝트 핵심 담당

### 4. 임원용 문서는 요약만 유지한다

- 임원 문서에는 개인별 세부 TASK를 다 적지 않는다
- 월간 문서에서만 개인별 배정을 상세 관리한다

## 추천 운영 프로세스

### 주간 운영

1. 각 팀 리더가 월간 파일에서 신규/변경 일정 반영
2. 프로젝트는 `EPIC` 기준으로 담당자와 기간 갱신
3. 운영/개선은 운영 테마별로 주간 TASK 묶음 반영
4. 배정률 80% 초과 인원, 미배정 인원, 병목 인원을 확인
5. 요약 결과만 `executive_overview.md`에 반영

### 월간 운영

1. 당월 파일 신규 생성 또는 갱신
2. 분기 파일과 월간 파일 정합성 점검
3. 종료된 프로젝트/운영 테마 정리
4. 다음 달 투입 예정 리소스 선배정

### 분기 운영

1. 분기 목표 프로젝트 확정
2. 운영/개선 상시 업무 묶음 재정의
3. 팀 간 인력 재배치 여부 검토

## 추천 데이터 표준

모든 일정 항목은 아래 속성을 갖게 한다.

| 항목 | 설명 |
|---|---|
| work_type | `project` 또는 `ops` |
| team | `기획`, `B2C개발`, `B2E개발` |
| owner | 담당자 |
| jira_key | EPIC 또는 TASK 키 |
| title | 업무명 |
| start | 시작일 |
| end | 종료일 |
| allocation | 배정률 |
| status | 예정 / 진행중 / 지연 / 종료 |
| priority | 상 / 중 / 하 |
| manager | 팀장 또는 리드 |

## 권장 역할 분담

- 임원: 우선순위와 리소스 재배치 결정
- 팀장/리드: 월간 파일 갱신 책임
- 구성원: 개인 파일 또는 입력 폼 기준 일정 요청/수정
- 자동화 에이전트: JIRA 반영, Mermaid 생성, 월간 파일 갱신

## 왜 이 방식이 적합한가

- 23명 규모에서는 ERP급 PPM 툴보다 Markdown 기반 경량 체계가 빠르다
- JIRA만으로는 개인별/팀별 용량 비교가 한눈에 잘 안 보인다
- Mermaid를 함께 쓰면 별도 BI 도구 없이도 시각성이 충분하다

다음 문서부터 바로 사용할 수 있는 템플릿을 제공한다.
